# Claude Code用 Phase 2実装指示書

`AGENTS.md`、`dev-thoroughness-checklist.md`、`docs/editor-model-refactor-plan.md`を最初に読み、Phase 2だけを実装してください。

過去に、正常系の1経路だけを実装し、別経路・データのライフサイクル・エッジ状態・保存再読込の不整合が後から判明しています。今回は以下の受入条件をすべて検証してから完了報告してください。

## 開始条件

Phase 1の次の修正が現在のコードとテストに含まれることを確認してください。欠けている場合はPhase 2へ進まず報告してください。

- `editorModelAuthority`が通常保存と`/api/preview-refresh`の両方で保持される
- 許可値が`legacy`と`editor`に限定される
- `editor`指定時、手動編集済みの`assets`と`imageCues`が再変換で上書きされない
- 保存、再読込、`migrate()`後も編集済みデータが不変

## Phase 2の目的

画像編集の正データを次の旧形式から移行します。

- `chapters[].image_cuts`
- `turn.cut`
- `review.json`の画像設定

移行後の正データは次です。

- `turn.id`
- `assets`: 動画全体で使える画像素材
- `imageCues`: セリフ区間に対する画像配置と表示設定
- `editorModelAuthority: "editor"`

Geminiの生成処理は従来どおり章単位で構いません。人間向けの編集UIでは章を意識せず、セリフ基準で画像を扱えるようにしてください。

## 絶対条件

- 動画生成結果とRemotion連携を壊さない
- `meta.json`の公開形式を変更しない
- Phase 3の`visualSegments`編集移行には着手しない
- 旧形式の読込互換を維持する
- フルリライトしない
- 新しい依存パッケージを追加しない
- ユーザーの既存未コミットファイルを変更しない
- ローカルテスト完了後にPhase 2だけをコミットして停止する

## 正データの切替

`editorModelAuthority: "editor"`への切替は、既存データの変換と検証が成功した保存時に行ってください。途中失敗で`legacy`と`editor`が中途半端に混在してはいけません。

切替後は次を守ってください。

- GET、通常保存、プレビュー更新のいずれでも旧形式から`assets`や`imageCues`を再生成しない
- UI操作では新形式だけを正として更新する
- 旧フィールドをUI操作の正データとして二重更新しない
- 旧フィールドは後方互換アダプターの入力としてのみ扱う

## UI要件

### 素材ライブラリ

- 章単位ではなく動画全体の`assets`を一覧表示する
- 素材追加、選択セリフへの配置、使用中か未使用かの判別ができる
- 同じ画像を複数箇所で使う場合、assetを複製せずimageCueを追加する
- 使用中のassetは無条件削除しない
- 削除拒否、または参照cueを含む明示削除のどちらかにする
- 未使用assetと壊れた`assetId`参照を区別する

### セリフ基準の画像編集

選択中セリフを基準に、次を操作できるようにします。

- 画像の配置と差し替え
- 非表示
- `fit`、`crop`、`filter`、`pad`、`bg`の編集
- cueの開始セリフと終了セリフの調整

章切替を画像編集の必須操作にしないでください。

### タイムライン

画像専用トラックを追加します。

- imageCueの区間を表示して選択できる
- 左右端ドラッグで開始・終了セリフを変更できる
- セリフ境界へスナップする
- 秒数の自由指定は追加しない
- `startTurnId`と`endTurnId`を正とする
- 横スクロール後もトラックヘッダーを固定する
- ドラッグ保存後もスクロール位置と選択状態を維持する

### 大演出との排他

画面全体を使う大演出と画像は同時表示しません。

- 大演出中もimageCue自体は削除しない
- 描画時だけ画像を抑制する
- Phase 3前のため、現在の大演出判定を利用する
- `visualSegments`の編集UI移行は行わない

## meta生成

`editorModelAuthority: "editor"`の場合は、`assets`と`imageCues`から既存meta形式へ解決してください。

- Remotion側の入力形式は変更しない
- 同じ入力ならlegacy経路と画像、区間、表示設定が等価になる
- cue境界で正しい画像に切り替わる
- 大演出区間で背後の画像が露出しない
- `legacy`の場合は従来経路を維持する

## 共通操作関数

UIイベントごとに配列を直接書き換えないでください。少なくとも次を共通化し、素材一覧・右ペイン・タイムライン・保存の全経路から利用してください。

- asset追加
- asset削除可否判定
- cue追加、移動、範囲変更、削除
- cueのasset差し替え
- turn削除・並べ替え時のreconcile
- 重複、孤立参照、範囲逆転のnormalize

## 実装前に列挙するテスト

以下をテストケース名へ落としてから実装してください。

- セリフ、asset、cueがそれぞれ0件
- 画像1枚または複数枚でcutアンカーなし
- 範囲外cut
- 同じassetを複数cueで使用
- 同じセリフに複数cue
- cueの開始と終了が同じ、または開始が終了より後
- 存在しないturnIdまたはassetId
- turn削除と並べ替え
- cue境界上のセリフ
- 章を跨ぐcue
- 大演出とimageCueの重複
- `hide: true`とassetIdの併存
- `fit/crop/filter/pad/bg`のnull、0、空文字
- editor authorityで旧フィールドだけが変更された状態
- legacy authorityで新フィールドが残った状態
- 通常保存とpreview-refresh保存
- 保存、再読込、再保存の冪等性
- サーバー再起動後の復元

`0`、`null`、未指定を同一視しないでください。特に`pad: 0`、crop座標の`0`、配列0件を欠損扱いしないこと。

## 必須検証

### 純ロジック

- legacyからeditorへの変換
- editorから既存meta相当への解決
- cueの追加、移動、削除、範囲変更
- asset参照整合
- turn変更時のreconcile
- 冪等性

### 保存ライフサイクル

- GET `/api/script`
- 通常保存
- `/api/preview-refresh`
- 保存後の再読込
- `editorModelAuthority`の保持
- 手動編集した`assets`と`imageCues`が再変換で消えないこと

### 実データ比較

`docs/story`の実データで、切替前後の次を機械比較してください。目視だけで完了にしないでください。

- 各セリフで選択される画像
- cueの開始位置と終了位置
- `fit/crop/filter/pad/bg/hide`
- metaへ出力される画像区間
- 大演出区間での画像抑制

## 完了前セルフレビュー

git diffを読み直し、次を確認してください。

1. 同じ操作をする別UI経路に処理漏れがない
2. 新旧形式の二重管理になっていない
3. editor切替後に旧形式から上書きされない
4. 通常保存とpreview-refreshで挙動が一致する
5. 追加だけでなく移動・削除・分割・並べ替えも整合する
6. null・0・空・孤立・重複・逆転を処理する
7. リロード後にUI状態とmeta結果が復元される
8. 既存テストを削除または弱体化していない
9. Phase 3の範囲へ踏み込んでいない
10. ユーザーの未コミット変更を含めていない

## 完了報告

次をすべて報告してください。

- コミットハッシュ
- 変更ファイル
- 正データと後方互換経路
- UIで可能になった操作
- 共通化した操作関数
- 追加テストのケース数と結果
- 既存回帰テスト結果
- 実データ比較結果
- 未実装事項と既知のリスク
- `git status`
- Phase 3へ進んでいないこと

単にテストが通っただけで完了とせず、UI操作から保存、再読込、meta生成までの一連のライフサイクルを確認してください。
