# safe-implement — 安全実装の手順

このリポジトリ（zundamon-video）で調査→方針→実装→検証を安全に行う手順。
共通ルール・壊してはいけない仕様は CLAUDE.md を必ず前提にする。

## 0. 依頼の解釈

- ユーザーが「方針だけ出して」「まだ編集しないで」と言ったら、**調査と方針提示で止まる**。編集しない
- 依頼内容から外れた「ついで修正」はしない。気づいた問題は方針の中で報告するだけ
- 曖昧な依頼は、実装前に1〜2個の具体的な質問で確認する（選択肢を添える）

## 1. 調査（関連ファイルだけ読む）

全体を読み直さない。症状・目的から対象を絞る:

| 症状・目的 | 最初に見る場所 |
|---|---|
| エディタUIの挙動・表示 | `story_editor.html`（`/usr/bin/grep` で関数名/ID検索→周辺のみ読む） |
| 動画の映像・演出・キャラ配置 | `video/src/StoryVideo.tsx` |
| 音声・話者・尺・読み | `make_story_audio.py`、必要なら `src/tts_voicevox.py` |
| 保存・取り込み・API・AIプロンプト | `story_editor.py` |
| シーン/表情/ポーズ編集 | `scene_editor.*` / `expression_editor.*` / `pose_editor.*` |

手順:
1. `/usr/bin/grep -n "キーワード" 対象ファイル` で行番号を特定（0件で不自然なら `LC_ALL=C /usr/bin/grep -a`）
2. Read の offset/limit でその関数の周辺だけ読む
3. **対になる処理を必ず確認する**: TSX側とHTML側のミラー実装、キャラとモブ、enterとexit。
   修正対象が片方だけに見えても、もう片方に同じロジックが無いか grep する

## 1.5 デバッグの型（バグ修正のとき）

**推測で直さない。原因を特定してから編集する。**

1. まず現象を説明できる仮説を挙げる（1つに飛びつかず、候補が複数あれば列挙）
2. 仮説は**安い順**に検証する: コードを読む → grepで反例を探す → 該当関数を単体で叩く。
   いきなり重い再現（レンダ・音声生成）に行かない
3. 「この行が原因」と言えるまで特定してから直す。特定できないまま直したくなったら、
   それは仮説であることを明示して、検証方法とセットで提示する
4. ユーザーの原因推測（「〜のせいでは?」）も鵜呑みにしない。事実確認してから同意/訂正する。
   実際に「VOICEVOXが原因」と見えて話者名の不一致だった前例がある
5. 同じアプローチで2回失敗したら、3回目の前に立ち止まって別仮説を探す。
   原因不明のまま試行錯誤を重ねない（CLAUDE.mdのルール通り報告して止まる）

## 2. 方針提示（編集前）

編集を始める前に、以下を短く提示する（自明な1行修正ならスキップ可）:

- 原因（コードの該当箇所を `file:line` で示す）
- 修正方針（どのファイルのどの関数をどう変えるか）
- 影響範囲（他に触る必要がある箇所・横展開先）
- 選択肢がある場合は推奨案を先頭に

「まだ編集しないで」と言われている場合はここで止まり、承認を待つ。

## 3. 実装ルール

- **最小差分**。依頼された修正に必要な行だけ変える
- 既存のファイル名・JSONキー・API パス・出力先・コマンド名を変えない
- 周辺コードのスタイル（命名・コメント密度）に合わせる。修正経緯コメントは書かない
- リファクタは依頼された時だけ。「ついでに整理」しない
- 新規依存パッケージは原則追加しない。どうしても必要なら実装前に理由を提示して確認
  （story_editor.py 系は標準ライブラリ縛り。video/ 側も小機能のための重い依存は不可）
- `story-01.json` をテスト用に書き換える場合は、必ず先にスクラッチパッドへ `cp` で退避し、
  検証後に復元して `git status` で確認する。`git stash` は使わない

## 4. 検証（段階制）

**必ず最も低いレベルから始める。変更内容に対して過剰な検証をしない。**

### Level 0: ドキュメント・コメントのみの変更
- テスト不要。差分を目視確認し「影響範囲なし」と報告

### Level 1: 1ファイル内の局所修正（文言・条件分岐・設定値）
- 構文チェックのみ:
  - Python: `python3 -c "import ast; ast.parse(open('FILE').read())"`
  - story_editor.html: `<script>`を抜き出して `node --check`
  - TSX: `cd video && npx tsc --noEmit -p .`
- 可能なら該当関数だけを `python3 -c` で直接呼んで最小確認
- フルテスト不要

### Level 2: 複数ファイル・JSON形式・パイプラインに触れる変更
- Level 1 の構文チェック + 以下から**関係するものだけ**:
  - Python変更 → 関係する回帰テスト（`python3 test_story_editor.py` 等・数秒）
  - StoryVideo.tsx の見た目変更 → `npx remotion still --frame=N` で対象フレーム1〜2枚
    （story-01.json に仮データが必要なら退避→編集→レンダ→復元）
  - エディタUI変更 → preview_start でブラウザ確認（ポートは空きに変更→終了後8771へ戻す）
  - 入出力形式に触れた → 変更前後のJSONを1件比較
- フルテストは原則不要

### Level 3: 広範囲・互換性に関わる変更（原則、事前確認済みのはず）
- Level 2 + 全Python回帰テスト3本 + `npx tsc`
- 主要フロー（保存→音声生成の判定部→プレビュー）を通し確認
- フル動画書き出し・音声全生成はこのレベルでも**ユーザーに理由を説明して合意後のみ**

### 共通ルール
- 長時間処理（動画書き出し・音声大量生成・VOICEVOX全ターン合成）は勝手に実行しない。
  必要なら「なぜ必要か・軽い代替は無いか」を先に提示
- テストできない場合（VOICEVOX未起動・実機依存等）は「未検証」と明示し、
  ユーザーが確認する手順を1行で書く

## 5. ターン終了前チェックリスト（毎回・30秒）

報告を書く前に必ず確認:

- [ ] `git status --short` — 意図しないファイルが変更されていないか
      （story-01.json / launch.json / テスト残骸が特に危険）
- [ ] 検証用の一時変更をすべて復元したか（ポート8771、退避したJSONの書き戻し）
- [ ] 「完了」と言う根拠になる**証拠**があるか（型チェック通過・テスト出力・スクショ・レンダ画像）。
      証拠なしに「動くはず」と報告しない
- [ ] 修正したバグと同種の箇所（ミラー実装・対の処理）を確認したか

## 6. 完了報告フォーマット

```
結論: ○○を修正した（or 方針提示のみ / できなかった）
変更: file:line ごとに1行
検証済み: 実行したチェックと結果
未検証: 実行しなかったものと理由（例: VOICEVOX起動が必要なため）
次にやること: （あれば。例: 音声生成ボタンで確認してください）
```

コミットはCLAUDE.mdのルールに従う（明示ステージ・story-01.json除外・日本語メッセージ）。
