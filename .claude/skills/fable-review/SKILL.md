# fable-review — 差分レビューの手順

直前の実装差分を、このリポジトリ（zundamon-video）の壊れやすいポイントを踏まえてレビューする。
全ファイルは読み直さない。**差分と、差分が触っている関連箇所だけ**を見る。

## 1. 差分の取得

```bash
git diff            # 未コミットの場合
git diff HEAD~1     # コミット済みの場合
git status --short  # 意図しないファイルが混ざっていないか
```

`git status` で以下が混ざっていたら即指摘:
- `video/public/story-01.json`（ユーザーの編集中データ。指示がない限りコミット対象外）
- `.claude/launch.json` のポートが 8771 以外のまま
- テスト用の仮データ・一時ファイルの残骸

## 2. チェックリスト（優先順）

### A. 依頼との一致（最重要）
- ユーザーの依頼と差分の内容が一致しているか
- 頼まれていない機能・リファクタ・「ついで修正」が混ざっていないか
- 依頼の一部が未実装のまま「完了」と報告されていないか

### B. 仕様破壊
- JSONキー・APIパス・出力ファイル名・CLI引数・ディレクトリ構成を変えていないか
- `_KNOWN_TURN_FIELDS`（story_editor.py）と実際に読み書きするフィールドがずれていないか
- 既存の台本データ（過去のstory-01.json）が新コードでも読めるか（後方互換）

### C. 横展開漏れ（このリポジトリ最頻出のバグ源）
- **TSX/JSミラー実装**: StoryVideo.tsx を変えたなら story_editor.html 側の対応ロジック
  （吹き出し計算・座標解決・登場退場判定）も更新されているか。逆も同様
- **キャラとモブ**: renderAvatar を直したら renderMob は？
- **enterとexit**、開始と終了、片方向と逆方向
- 新演出追加なら `docs/new-effect-checklist.md` の全項目
  （AI生成プロンプト文面・EFFECT_SETTING_SPECS・_KNOWN_TURN_FIELDS）

### D. 品質
- エラー処理が握りつぶしになっていないか（原因が伝わらない固定文言にすり替えていないか）
- ユーザー向けメッセージ・ログが日本語として分かりやすいか
- コメントが「修正の経緯」「何をしているか」の説明になっていないか（「なぜ」だけが正）
- 変数名・スタイルが周辺コードから浮いていないか

### E. 検証の適切さ（不足も過剰も指摘）
- 変更規模に対して検証が不足していないか
  （例: TSX変更で `npx tsc` すら未実行 / 見た目の変更で静止画確認なし）
- 逆に、過剰でないか
  （例: 文言修正でフルテスト / 低リスク変更で動画書き出しを要求）
- テスト用の一時変更（story-01.json・launch.json）が復元されているか
- 報告に「未検証項目」が正直に書かれているか
- 「完了」の主張に**証拠**（テスト出力・型チェック結果・レンダ画像・DOM確認）が伴っているか。
  「動くはず」だけの報告は要修正扱い

### F. 次のアクション
- ユーザーが次に何をすべきか（確認手順・実行コマンド）が明確か

## 3. 関連箇所の読み方

- 差分の関数を呼んでいる箇所: `/usr/bin/grep -n "関数名" 対象ファイル` で呼び出し元だけ確認
- ミラー実装の有無: 同名・類似名を両ファイル（story_editor.html / StoryVideo.tsx）で grep
- それ以上は読まない。全ファイルレビューはしない

## 4. 検証の追加実行

- レビュー中に不安が残る場合のみ、safe-implement の検証レベル表に従い**最小の検証を1つ**実行
  （構文チェック・該当テスト1本・静止画1枚まで）
- 重い検証（フルテスト・動画書き出し）を要求する場合は、なぜ必要かを必ず書く。
  低リスク変更には要求しない

## 5. レビュー結果の出力

重要な指摘から順に、簡潔に:

```
判定: 問題なし / 要修正 / 要確認
[重要] 仕様破壊・依頼不一致（あれば。file:line + 理由 + 修正案）
[中] 横展開漏れ・品質（あれば）
[軽微] スタイル等（あれば。省略可）
検証の評価: 不足/過剰/適切のどれか + 根拠1行
```

- 「問題があれば修正して」と言われている場合: [重要]と[中]は修正まで行い、
  修正後に safe-implement の該当レベルの検証を再実行して報告する
- 指摘ゼロなら「問題なし」と一言で終える。無理に指摘を作らない
