# 現行の手直し済み台本から抽出した生成品質基準

対象は `video/public/story-01.json`（2026-07-18時点の作業中V2台本）。AI出力を手作業で改善した結果を、今後のV2台本生成プロンプトへ反映するために集計した。文字数は空白と改行を除く。

## 定量サマリー

| 項目 | 実測 | 生成時の基準 |
| --- | ---: | --- |
| 全体 | 158ターン / 3,900文字 / 616.3秒 | 指定なしでは8〜11分、120〜170ターン、3,200〜4,300文字 |
| 密度 | 1分あたり380文字 / 15.4ターン | 350〜420文字 / 13〜17ターン |
| 1ターン | 平均24.7文字、中央値25文字、90%点40文字 | 通常13〜36文字。説明・宣言だけ37〜55文字 |
| 視認性 | 70ターンに手動改行。分割行の中央値18文字 | 意味の切れ目で12〜22文字。台詞の約半数で使う |
| 音声尺 | 1ターン平均4.16秒、1文字あたり約0.158秒 | 文字数だけでなく、反応・間・画面行動を入れて呼吸を作る |
| 会話リズム | 話者交代49回。連続話者は2〜5ターンが中心 | 一問一答に固定せず、行動や感情を積む短い連続を許す |

空白だけのターンは9件で、場面開始・未来視・時間経過などの映像上の間に限定されている。通常の尺合わせには使われていない。

## 物語構成の特徴

1. 冒頭から主人公が具体的に動いており、説明で始めない。
2. 原因は回想と未来視で映像化し、後半に失敗を二段階（婚活、地域活動）で悪化させる。
3. めたんが原因を言い切るだけでなく、ホワイトボードで「推し活そのものではなく後回し癖」という根本原因を整理する。
4. 解決は大きな人格改造でなく、母への電話・友人への連絡という小さな一歩から始める。
5. 最後は未来の酒場で結果を見せ、財布のオチで本編を閉じてからCTAへ移る。

このため、標準形は「フック → 原因の可視化 → 失敗/悪化を2段階 → 根本原因 → 小さな改善 → 結果と軽いオチ」とする。題材・指定尺に応じて各段は縮めるが、失敗と改善が同じ場面で即座に終わらないようにする。

## 表情・ポーズ・舞台操作

- `stage.update`: 136/158ターン（86%）。セリフの内容と表情・ポーズを対応させている。
- 表情の上位は normal 38、trouble 34、happy 23。驚き・怒り・焦り・泣きは山場に集中し、通常表情だけの単調さを避けている。
- ポーズの上位は proud 29、idle 29、droop 14、point 11、cheer 9。得意げな暴走、落ち込み、説明、喜びが判別できる。
- `stage.enter` は17件、`exit` は1件。人物を会話の都度出し入れせず、シーン開始または登場の必然がある時だけ操作している。

生成では、登場済みの人物だけを更新する在席簿ルールを守ったうえで、感情または行動のビートが変わる箇所に `stage.update` を置く。ポーズは利用可能な値のみを使い、スマホでの通話・送信には `smartphone` を選べるようにする。

## カメラと場面転換

- `framing`: sceneDefault 79、speaker 59、slot 2。話者寄りと引きの状況説明を使い分けている。
- `cameraMotion`: 25ターン（16%）。内訳は zoom 24、pan 9、shake 3、tilt 1で、zoomを主に決意・不安・ツッコミに使う。
- `transition`: 13件。fade-white 5、fade-black 3、slide-left 2、wipe-left 2、cut 1。ほぼ場面または時間の切替に対応している。

カメラ移動は常時動かさず、感情や行動に理由を持たせる。特にshakeは驚き・衝突の瞬間だけ、tiltや手動構図は必要な例外カットだけにする。

## 演出・画面表示・音

- `effects`: flashback 24、visionNoise 4、impactLines 2、zoomPunch 1、irisOut 1。回想は連続区間として維持し、未来視にはvisionNoiseを重ねている。
- `caption`: 32ターン。回想中の「昨晩 自宅」と未来視中の「未来のずんだもん」を連続表示し、時間軸を迷わせない。
- `displayMode`: standard 149、whiteboard 5、zunAi 4。特殊表示は通知の提示と、根本原因の説明に限定される。
- SEは6ターン・9音。送信、通知、場面開始といった見える動作にだけ同期している。

したがって、演出は「使えるから使う」のではなく情報を映像化する必要がある時に限定する。ホワイトボードは3項目を段階表示できる説明の転換点、ZunAI等はログや通知の内容自体が物語の証拠になる場面に使う。

## 実装への反映

`story_editor.py` の `_build_script_prompt_v2()` に、上記の長さ、構成、吹き出し、表情・ポーズ、カメラ、演出、特殊表示、空ターンの基準を追加した。既定の長さも実測に合わせて更新し、既存ポーズ一覧に `smartphone` を加えた。JSONスキーマ、API、レンダリング仕様には変更を加えていない。
