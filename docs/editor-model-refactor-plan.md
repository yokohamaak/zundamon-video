# 人間向け編集モデルへの移行計画

## 1. 結論

Geminiの生成形式と、人間がレビュー画面で扱う編集形式を分離する。

- Geminiは従来どおり章単位で台本・画像候補・演出案を生成する。
- 章は物語構造、見出し、ページめくり、intro/trivia/outroの境界として残す。
- 画像と大演出は章の所有物にせず、セリフへアンカーされた独立データへ変換する。
- `meta.json`とRemotionの入力形式は当面変更しない。
- 既存データは読み込み時に変換でき、移行後は人間向け編集モデルを正とする。

目的は章跨ぎを積極的に使うことではない。AI生成上の都合を編集UIへ漏らさず、ユーザーが「どのセリフから何を表示するか」だけを扱えるようにすることである。

## 2. 現状の問題

現在は以下が章に集約されている。

- `chapters[].image_cuts`: 画像素材と検索情報
- `turn.cut`: 章内画像番号
- `chapters[].vizList`: 大演出の内容
- `turn.vizSeg`: 章内ローカルな演出ID
- `review.json`: 章番号・カット番号をキーにした画像設定

この構造はGemini生成、章単位再生成、画像一括取得には都合がよい。一方、編集時には以下の問題がある。

- 他章の画像を自然に再利用できない。
- 画像を変えるために章内カット番号を意識する必要がある。
- 大演出一覧の番号が章内番号になり、動画全体の構造と一致しない。
- 章境界の変更が画像・演出の所属に影響する。
- UI上のタイムラインと保存モデルの境界が異なる。

## 3. 編集モデル

既存`script.json`を引き続き編集データの保存先とし、トップレベルへ以下を追加する。別の編集ファイルを増やして二重の正を作らない。

```json
{
  "schemaVersion": 2,
  "chapters": [],
  "script": [],
  "assets": [],
  "imageCues": [],
  "visualSegments": []
}
```

### 3.1 chapters

章は軽量な構造メタとして維持する。

- `id`
- `title`
- `summary`
- `section`
- `hook`
- `confidence`
- `source_hint`

`image_cuts`と`vizList`は移行元として後方互換読み込みを行うが、新UIの正規保存先にはしない。

### 3.2 script

各セリフへ永続的な`id`を追加する。

```json
{
  "id": "turn-0001",
  "speaker": "四国めたん",
  "text": "...",
  "chapter": 1
}
```

- IDは一度付与したら分割・保存・再読込で変えない。
- セリフ分割時は前半が元IDを維持し、後半へ新IDを発行する。
- 配列indexはアンカーとして保存しない。

### 3.3 assets

画像ファイルそのものと、素材固有の情報を管理する。

```json
{
  "id": "asset-0001",
  "file": "ch_01_00.jpg",
  "query": "横浜駅 構内",
  "queryJa": "横浜駅の構内",
  "kind": "ambient",
  "attribution": "...",
  "sourceChapter": 1
}
```

素材側に置くもの:

- ファイル名
- 検索語
- 種別
- 出典
- 生成元の章タグ
- 画像サイズなどの素材情報

`sourceChapter`はフィルター用であり、所有権ではない。他章でも利用できる。

### 3.4 imageCues

画像を「このセリフから切り替える」というキューとして配置する。画像は次のキューまで継続する。

```json
{
  "id": "image-cue-0001",
  "turnId": "turn-0004",
  "assetId": "asset-0001",
  "fit": "cover",
  "crop": null,
  "filter": null,
  "pad": 0,
  "bg": null,
  "hide": false
}
```

配置側に置くもの:

- 使用開始セリフ
- crop / fit / filter / pad / bg / hide

同じ素材を複数箇所で別クロップにできるよう、表示調整値はassetではなくcueに置く。

操作規則:

- 選択セリフがキュー開始位置なら、素材クリックでそのキューを差し替える。
- キュー開始位置でなければ、そのセリフから新しいキューを追加する。
- キュー削除時は直前の画像を引き継ぐ。
- 動画先頭にキューがなければ画像なしとする。
- 章境界では画像を自動リセットしない。
- Gemini生成結果の初期変換時だけ、従来どおり章頭・`turn.cut`変化位置へキューを置く。

### 3.5 visualSegments

大演出を動画全体の独立セグメントとして管理する。

```json
{
  "id": "visual-0001",
  "type": "panel",
  "startTurnId": "turn-0010",
  "endTurnId": "turn-0013",
  "mode": "layout",
  "config": {},
  "keyframes": []
}
```

`mode`:

- `replace`: 通常画像を完全に隠す。比較、全面カードなど。
- `layout`: 現在画像を素材としてレイアウトを変更する。解説パネル。
- `overlay`: 通常画像の上へ重ねる。数字、注釈、クイズ。

制約:

- `replace`と`layout`は同じ時間帯に1つまで。
- `overlay`は通常画像との併用を許可する。
- 章跨ぎを保存形式では禁止しない。必要ならUIの既定操作で章内へ収める。
- IDは動画全体で一意にする。

## 4. 描画規則

表示優先順位:

```text
replace
layout
overlay
通常画像(imageCue)
```

画像キューは大演出中も裏で継続する。`replace/layout`終了後は、その時点で有効な画像へ戻る。大演出の前後で画像キューを自動分割しない。

章めくりトランジションは生画像ではなく、章末直前の合成済みVisualSurfaceを対象にする。

- 通常画像
- replace/layout演出
- overlay演出

を含む完成面を`chapterBoundary - 1 frame`で固定し、その面全体へ3D回転を適用する。短期的な代替として、合成面を実装できるまで`replace/layout`継続中の章切替だけクロスフェードにしてもよい。

## 5. UI

### 5.1 画像タブ

- 現在位置で有効な画像と、その開始セリフを表示する。
- 「この章の素材」を既定フィルターにする。
- 「全素材」「使用中」「未使用」へ切り替えられる。
- 他章の素材も選択できる。
- 素材削除と配置削除を分ける。
- 共有素材の削除時は使用箇所を表示し、参照中なら不用意にファイル削除しない。

### 5.2 タイムライン

画像トラックを追加する。

```text
画像  [横浜駅────────][構内図────][工事写真────]
大演出      [パネル────]     [比較────]
```

- 画像ブロックの開始はセリフ単位でドラッグできる。
- `replace/layout`中は通常画像ブロックを薄くし、「設定済みだが演出で隠れている」と示す。
- 選択セリフ、画像タブ、タイムラインを双方向同期する。

### 5.3 大演出タブ

- 章内の`1.クイズ / 2.パネル`一覧を廃止する。
- 選択セリフに有効な大演出を表示する。
- 無ければ「＋大演出」を表示する。
- 全体構造と範囲調整はタイムラインを正とする。

## 6. 既存データの移行

移行は冪等にする。同じファイルへ複数回適用してもIDやキューが増殖しないこと。

### 6.1 turn ID

- ID無しセリフへ順番に安定IDを付与する。
- 既存IDは変更しない。

### 6.2 assets

- `chapters[].image_cuts`と`review.json`を突き合わせてassetを作る。
- 初期IDは章・cutから決定的に生成してよい。
- ファイル名は初期移行で変更しない。

### 6.3 imageCues

- script順に`(chapter, cut)`を走査する。
- 先頭と、直前から組が変わったセリフへキューを作る。
- 対応するreview設定をcueへコピーする。
- 同じ画像が再登場してもassetは共有し、cueは別にする。

### 6.4 visualSegments

- `chapters[].vizList`をトップレベルへ移す。
- `chapter + vizSeg`からグローバルに一意なIDを作る。
- 所属セリフの先頭・末尾をturn IDアンカーへ変換する。
- フラグと`vizPoints`をkeyframesへ正規化する。
- 既存形式は読み込み互換として残す。

## 7. ライフサイクル整合

実装前に全経路を列挙し、共通関数を必ず通す。

- 追加
- 差し替え
- 移動
- 削除
- セリフ分割
- セリフ削除
- 章境界変更
- 保存
- 再読込
- 旧形式移行
- meta-only更新
- 音声再生成

最低限の規則:

- キュー位置のセリフを分割: キューは前半IDに残る。
- キュー位置のセリフを削除: 次のセリフへ移す。次が無ければキューを削除する。
- 大演出の端点セリフを削除: 残存範囲へ縮め、空なら演出を削除する。
- asset削除: 参照中cueがあれば拒否または明示的な一括解除を要求する。
- cue削除: assetファイルは削除しない。
- ID再利用は禁止する。
- `pos:0`と未指定を区別する。

## 8. meta / Remotion境界

当面は`meta.json`の公開形式を維持する。

- `main_story.py`内部でassets/imageCues/visualSegmentsを絶対時刻付きtopicsへ解決する。
- Remotionは従来の`meta.topics`と`meta.script`を読み続ける。
- 既存プロジェクトを新形式へ移行し、編集せずmeta生成した場合に描画結果が同等であることをテストする。
- 新形式が無ければ旧`chapters[].image_cuts`・`vizList`へフォールバックする。

## 9. 段階的実装

### Phase 1: スキーマと変換のみ

- turn ID
- assets
- imageCues
- visualSegments
- 旧形式からの冪等変換
- 保存・再読込
- UIとmeta出力は変更しない

### Phase 2: 画像編集をimageCuesへ移行

- 素材ライブラリ
- セリフから画像キュー追加・差し替え・削除
- 画像タイムライントラック
- placement単位のcrop等
- meta生成をimageCues対応

### Phase 3: 大演出をvisualSegmentsへ移行

- 章内一覧を撤去
- 選択セリフ基準の右ペイン
- visualSegmentsとkeyframesを編集
- 旧vizList読み込み互換

### Phase 4: 合成面トランジション

- VisualSurfaceを共通化
- 章末完成面をページめくり対象にする
- replace/layout/overlayの描画優先順位を統一

各Phaseを独立コミットにし、次へ進む前に既存テストと手動確認を完了する。

## 10. 必須テスト

実装前にケース名を列挙してからコードを書く。

### 変換

- 旧形式からassets/imageCues/visualSegmentsへ変換
- 変換を2回実行しても結果不変
- 章ごとに同じcut番号・vizSeg IDがあっても衝突しない
- 未取得画像、画像なし、空章

### imageCues

- 先頭キュー
- 途中キュー
- 章境界を越えて画像継続
- 同じassetを複数cueで共有
- cue削除で直前画像を継承
- cue位置のセリフ分割・削除
- crop等がassetでなくcueごとに独立
- 画像差し替え後の保存・再読込

### visualSegments

- 追加・移動・削除・範囲変更
- 端点セリフの分割・削除
- replace/layoutの重複拒否
- overlayとの併用
- 章境界を跨ぐ範囲
- keyframeの追加・移動・削除・pos:0

### meta / 描画

- 無編集移行前後でmetaの意味が同等
- imageCueが次のcueまで有効
- replace/layout中に通常画像が露出しない
- 演出終了後に有効画像へ復帰
- 章めくり中に裏画像が一瞬露出しない
- 保存してプレビュー更新
- 再読込後のpreviewState

## 11. 実装上の禁止事項

- Gemini生成形式を先に変更しない。
- `meta.json`とRemotion公開契約を一度に変更しない。
- 旧データをその場で破壊しない。
- indexや章ローカル番号を永続アンカーにしない。
- assetと配置設定を再び同じ辞書へ混在させない。
- UI経路ごとに検証・整合処理を重複実装しない。
- 一括置換によるフルリライトをしない。
- 従量課金APIや外部サービスを追加しない。

## 12. 完了条件

- Gemini生成パイプラインは従来どおり動く。
- 旧プロジェクトを開いて無損失で編集モデルへ移行できる。
- 人間は章番号・cut番号・章内演出番号を意識せず編集できる。
- 画像はセリフから切り替え、他章素材を再利用できる。
- 大演出は動画全体の独立セグメントとして扱える。
- meta生成とRemotion描画に回帰がない。
- 全経路、全ライフサイクル、エッジ状態、再読込のテストがある。
