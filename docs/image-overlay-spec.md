# 画像オーバーレイ機能 設計メモ

ステータス:
- 方針整理用メモ
- まだ未実装

目的:
- 参考画像、スクリーンショット、強調画像などを動画上に自由配置して重ねられるようにする
- 既存の `insert`（全画面情報演出）とは別扱いにし、軽量な視覚補助レイヤーとして扱う

---

## 結論

画像オーバーレイは **`insert` ではなく独立した `overlay` レイヤー** として設計する。

理由:
- `insert` は全画面UI演出であり、役割が違う
- 画像オーバーレイは複数同時表示したい
- 位置と大きさを自由配置したい
- ターン途中から途中まで出したい
- `insert` と同時表示したい
- セリフ跨ぎで使いたい

つまり分類はこうする。

- `insert`
  - 全画面の情報演出
  - 例: `warning / ok / chat / teamchat / mailer`
- `overlay`
  - 画面に重ねる補助画像
  - 例: スクショ、参考画像、図、強調画像

---

## 要件整理

ユーザー合意済みの前提:

- 複数枚同時表示したい
- 位置は D&D で自由配置したい
- サイズはドラッグで変更したい
- 表示タイミングは途中から途中まで可能にしたい
- `insert` と同時表示したい
- 用途は「小さな参考画像」と「大きな強調画像」の両方
- セリフ跨ぎを許容したい

---

## データモデル方針

### 保存先

`turn.overlays[]` ではなく、**`story.overlays[]` を新設**する。

理由:
- ターン内限定だと説明パートで不便
- セリフを跨いだ表示を自然に扱いたい
- タイムライン上で1本の区間バーとして扱いやすい

### 時間の持ち方

絶対秒ではなく、**ターンアンカー + オフセット秒** で持つ。

理由:
- 音声再生成で `start/end` が多少動いても追従しやすい
- 既存の `turn.start/end` に寄り添える
- UI上も「どのターン基準で始まるか」が把握しやすい

### 初期スコープ

- アンカー粒度は **turn + 秒** まで
- `sentence` 単位アンカーは最初は不要

理由:
- UIがかなり簡単になる
- まず欲しい用途は十分カバーできる

## 提案JSONスキーマ

```json
{
  "overlays": [
    {
      "id": "ov-0001",
      "kind": "image",
      "src": "overlays/example.png",
      "x": 0.72,
      "y": 0.32,
      "w": 0.28,
      "opacity": 1,
      "z": 3,
      "start": {
        "turnId": "turn-0010",
        "at": 0.4
      },
      "end": {
        "turnId": "turn-0013",
        "at": 1.2
      }
    }
  ]
}
```

### 各フィールド

- `id`
  - オーバーレイ識別子
- `kind`
  - まずは `"image"` 固定でよい
  - 将来的に `text / arrow / badge` へ拡張可能
- `src`
  - `video/public/` から参照できる画像パス
- `x`, `y`
  - 画面基準の正規化座標（0..1）
- `w`
  - 画面幅に対する比率
  - 高さは画像の縦横比で自動計算
- `opacity`
  - 0..1
- `z`
  - 前後順
  - 大きいほど前面
- `start.turnId`, `end.turnId`
  - 開始/終了の基準ターン
- `start.at`, `end.at`
  - そのターン開始からの秒オフセット

---

## 座標系

初期実装では **`screen` 基準のみ** にする。

意味:
- 画面全体に対して固定された座標に置く
- カメラズームや話者寄りに影響されない

理由:
- 「参考画像を右上に出す」「図を左下に置く」といった用途に向く
- UIとして理解しやすい
- 実装が軽い

将来候補:
- `space: "stage"` を追加
  - 背景やキャラに貼り付くような表現向け

ただし初期実装では不要。

## 重なり順

### 描画ルール

- 基本は `z` の昇順で描画
- 同じ `z` の場合は **配列の後ろが前面**
- UI上の理解としては **あと勝ち**

### 理由

- ユーザーの直感に合う
- タイムラインでも扱いやすい
- 実装が単純

---

## insertとの関係

`overlay` は `insert` と **同時表示可能** にする。

ただしレイヤー順は最初に決めておく。

おすすめ:

1. 背景・キャラ
2. `overlay`
3. 吹き出し
4. `insert`
5. 回想/フェード等の全画面演出

理由:
- `insert` は全画面演出なので、基本はオーバーレイより上に出る方が自然
- オーバーレイは通常画面上の補助情報として扱う

補足:
- 将来「insertの上にも画像を出したい」が出たら、その時点で `layer: "normal" | "over-insert"` を増やす
- 初期実装では不要

## プレビューUI

### 画像の配置

プレビュー上で直接編集する。

できること:
- ドラッグで移動
- 角または辺をドラッグして拡大縮小
- 選択中オーバーレイに枠を表示

### サイズ変更

仕様:
- 端や角のハンドルをドラッグして変更
- 初期実装では **縦横比固定**

理由:
- 画像が潰れにくい
- UIが単純
- 多くの用途で十分

将来候補:
- Shift押し等で縦横比解除

初期実装では不要。

## タイムラインUI

### 基本方針

オーバーレイ専用レーンを追加し、**1オーバーレイ = 1本の区間バー** として表示する。

### なぜ専用レーンか

- BGMと違い、複数枚重なることがある
- セリフやBGMと分けた方が見やすい
- 開始・終了・移動の操作対象が明確になる

### レーン表示案

- セリフレーン
- BGMレーン
- Overlayレーン

Overlayレーン内では、重なるバーを自動で段積みする。

### 段積みルール

- 同時刻に重なるオーバーレイ同士は、上から順に別段へ積む
- 段数は自動計算
- 選択中バーは枠強調

### バー操作

- バー本体ドラッグ
  - 全体移動
- 左端ドラッグ
  - 開始時刻調整
- 右端ドラッグ
  - 終了時刻調整
- クリック
  - 選択

### バー表示情報

初期は最低限でよい。

- ラベル
  - ファイル名末尾、または `img-1` 等
- 色
  - Overlay専用色
- 透明度
  - 固定

## 編集フロー

おすすめフロー:

1. 選択中ターン付近で「画像オーバーレイ追加」
2. 画像ファイルを選択
3. プレビュー上で位置・サイズ調整
4. タイムラインで開始・終了を調整

補足:
- 追加時の開始位置は現在ターン付近
- 追加時の終了位置は `+3〜5秒` の仮値でよい

## 画像の登録元

初期実装では **既存ファイル選択のみ** にする。

候補:
- `video/public/overlays/`
- または `preview-assets` で配信可能な画像群

理由:
- アップロード機能は実装コストが増える
- まずは既存画像で運用できる形を優先したい

将来候補:
- エディタからの画像追加
- `video/assets/overlays/` へのコピー支援

初期リリースではここまでで十分。

## 判断メモ

- `insert` に寄せない
  - 役割が違うため
- `turn` 所属にしない
  - セリフ跨ぎ用途に弱いため
- `absolute seconds` にしない
  - 音声再生成時のズレに弱いため
- `sentence` アンカーにしない
  - 初期実装としては過剰なため

---

## 現時点の推奨結論

- 画像オーバーレイは **独立レイヤー**
- データは **`story.overlays[]`**
- 時間は **`turnId + at秒`**
- 位置は **screen基準**
- 大きさは **プレビュー上でドラッグ調整**
- 時間は **タイムラインのOverlayレーンで調整**
- 重なり順は **あと勝ち**
